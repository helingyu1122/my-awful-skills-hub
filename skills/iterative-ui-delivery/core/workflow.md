# 工作流

## 标准流程

1. **Mini Brief（3 行）**
- 本轮目标：只写一个可观察改进
- 用户动作：本轮用户能完成什么操作
- 验收信号：看到什么现象算完成

2. **小步实现**
- 只改一个垂直切片（页面区域/交互块）
- 非阻塞问题不做大重构

3. **快速评审**
- 立即展示结果
- 把口语反馈改写为“可执行改动项”

4. **增量修正**
- 仅修改受影响模块
- 未被点名的行为默认保持不变

5. **每轮 Gate（强制）**
- Build Gate：`npm run build`
- UX Gate：样式一致性（间距、颜色、组件、字体、状态）
- Behavior Gate：变更交互端到端可用
- Boundary Gate：目录边界清晰（app/pages/features/shared）

## 关键判断点

- 反馈是否可执行：不可执行就先改写再开发
- 变更是否过大：过大就继续拆轮次
- 是否引入风格漂移：有漂移先对齐再继续功能

## 输出约定

1. 先给本轮目标与完成状态
2. 再列具体改动文件与行为变化
3. 最后给下一轮建议（1~3条）
